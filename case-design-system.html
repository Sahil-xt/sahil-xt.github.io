Case study 03 · Design System

From inconsistency to a shared design language

RoleLead UX Designer
Team4 designers + engineering
ApproachAudit → Identify → Systematise → Document
The situation

4 designers, 15+ modules, no shared library

This project didn't follow a traditional UX design process — it was a systems problem that needed a systems response. Four designers were working in parallel on different modules with no shared components, no naming conventions, and no agreed-upon patterns. Stakeholders kept flagging inconsistencies in demos. New features took longer to ship because every team reinvented the same patterns.

The approach: Audit → Identify patterns → Build the system → Document → Maintain. We adopted Atomic Design (atoms → molecules → organisms → templates → pages) as our framework, but the real work was in the audit and pattern-finding that came before.

Audit

Dumping everything onto one Figma canvas

The first step was a full product audit — every screen from every module dumped into one Figma canvas. Seeing the whole product at once was the fastest way to spot patterns and inconsistencies that weren't visible when working module-by-module.

From the canvas audit, we ran a team workshop to organise findings under five consistency pillars:

Key discovery

One pattern hiding across six modules

The most valuable find from the audit: every module that involved creating a policy — attendance, leave, holidays, shifts, WFH, expense — followed the same logical structure even though the UI looked completely different each time.

Every policy creation flow had three parts: policy header (name + description) → policy parameters (module-specific rules) → applicable to (employee assignment). Standardising this single pattern eliminated a whole class of inconsistency across six modules in one decision.

Why this mattered: Once we named the pattern, engineers and designers could reference it. "This is a policy creation flow" became a shared vocabulary that didn't exist before. Naming things is underrated.
Building the library

Atomic structure, Figma variants, shared tokens

We inventoried all UI components: input controls, navigation elements, informational components, branding tokens. Then categorised them using Atomic Design — an early categorisation mistake (mapping components to wrong atomic levels) was caught and corrected through team review, and we documented the decision so it wouldn't repeat.

  • Variants: Component states (default/hover/disabled/error) as Figma variants — not separate components
  • Nested components: Update one atom and it propagates through all molecules and organisms
  • Shared token layer: Web and mobile reference the same token values — "primary action" has consistent meaning across platforms
  • Code snippets: CSS/XML generated per component to reduce handoff ambiguity
  • Version control: Figma branches for major updates, allowing rollback when a component change broke existing designs

Documentation & outcome

A wiki, not just a Figma file

The Figma library was only half the system. A documentation wiki covered all five consistency pillars — not just what components look like, but why they're designed that way and when to use them. Icon-to-module mapping (fingerprint = attendance, umbrella = leave, hand-coins = payroll) was documented here, ensuring IA consistency across web and mobile navigation.
