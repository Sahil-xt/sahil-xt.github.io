Case studies

Four projects from my time at Mewurk — an all-in-one HRMS platform for Indian SMEs. Each shows a different challenge: hardware UX, system-level design, employee trust, and cross-platform adaptation from web to mobile.

Web to mobile — a thread across all projects
Across all modules, I was responsible for translating web designs into the Mewurk mobile app — rethinking information hierarchy, touch interactions, and context-of-use for employees on the go. This wasn't just resizing; it meant redesigning flows that worked for a factory worker checking in on a phone with one hand.
Case study 01 · Attendance & Kiosk
Face scan attendance — from mobile chaos to hardware kiosk
Redesigning how 500k+ employees clock in every day — and extending it to a dedicated face-recognition kiosk for factory and field settings.
Lead designer Web + Mobile Hardware kiosk
The problem
HR managers were spending hours reconciling attendance records. Employees were gaming location-based check-ins. Factory floors needed a touchless, zero-training solution for workers with low digital literacy.
My role
I led UX end-to-end — discovery with HR managers and field employees, shipped mobile flows, the admin web dashboard, and the kiosk interface. I worked closely with the PM and hardware engineering team throughout.
Web to mobile
The admin dashboard was designed for web first — dense data tables with filters and drill-downs. For mobile, I stripped it to the essentials: today's summary, flagged exceptions, and one-tap approvals. Managers didn't need full reports on the go — they needed to act fast.
Key decisions
The kiosk UI had to work for workers with no smartphone experience. I stripped it to a single action per screen, used large visual feedback states (green/red face frame), and replaced all text instructions with iconography and audio cues. On admin, I redesigned the dashboard to surface exceptions first — late arrivals, missed punches, anomalies — rather than raw logs.
What shipped
A face-scan kiosk requiring zero training, deployed across 1000+ SME customers. Mobile attendance with geo-fencing, selfie verification, and field visit tracking in a single flow.

Case study 02 · Leave & Holidays
Designing leave management that works for everyone — not just HR
Leave is one of the most emotionally loaded features in any HRMS. Getting it wrong affects trust, morale, and compliance. I led the full design — policy configuration for admins and a transparent, frictionless experience for employees.
Lead designer Web + Mobile Policy-driven UX
The problem
SMEs in India have highly varied leave policies — different accrual rules, carry-forward limits, holiday calendars by location, and approval chains. The existing approach was too rigid for admins and too opaque for employees, who often didn't know their balance or why a request was rejected.
My role
I led design for both the admin configuration side and the employee-facing experience — running separate research sessions with HR managers and employees to understand the gap between how policies were set and how they were experienced.
Web to mobile
On web, admins needed rich policy configuration — leave types, accrual rules, holiday calendars, approval chains. On mobile, employees needed the opposite: a clear balance, a simple apply flow, and real-time visibility into approval status. I redesigned the mobile apply flow to be under 3 taps, with plain-language policy summaries inline so employees always knew what they were entitled to.
Key decisions
The hardest design challenge was the policy builder — admins needed flexibility without complexity. I introduced a progressive disclosure approach: sensible defaults upfront, with advanced rules tucked behind an "advanced settings" toggle. For employees, I added a leave calendar that showed team availability alongside personal balance — reducing the back-and-forth with managers about "when is a good time."
What shipped
A flexible leave policy builder supporting multiple leave types, accrual rules, and location-specific holiday calendars. An employee mobile experience with a clear balance summary, 3-tap apply flow, real-time approval tracking, and inline policy explanations.
Case study 03 · Design system
Building Mewurk's design system from the ground up
As the product scaled to 15+ modules across web and mobile, inconsistency was slowing down design and engineering. I led the effort to audit, document, and ship a shared system.
Lead designer Web + Mobile Pattern library
The problem
With 15+ modules built over time by multiple hands, the product had accumulated UI debt — inconsistent spacing, duplicate components, no shared language between design and engineering. New features took longer to ship as teams reinvented patterns each time.
My role
I owned the design system initiative — auditing existing UI across web and mobile, identifying recurring patterns, defining component specs, and aligning with engineering on implementation. This ran alongside active product work, not as a dedicated project.
Web to mobile
One of the core challenges was that web and mobile had evolved independently — different spacing scales, different component names, different interaction patterns for the same actions. I created a shared token layer that both platforms could reference, so a "primary action" meant the same thing everywhere even if it rendered differently.
Key decisions
Rather than building a perfect system first, I took an incremental approach — documenting what existed, standardising the most-used patterns first (forms, tables, modals, notifications), and letting the system grow from real usage. I prioritised the admin-facing web UI first since that's where most complexity lived, then mapped each component to its mobile equivalent.
What shipped
A Figma component library covering core UI patterns for web and mobile, a pattern documentation wiki shared with engineering, and a naming convention adopted across all new modules. Reduced design-to-handoff time noticeably for new features.
Case study 04 · Payroll & Field visits
Making payroll transparent and field tracking non-intrusive
Two modules with the same core tension — powerful for admins, anxiety-inducing for employees. The challenge was building trust on both sides of the product.
Lead designer Web + Mobile Trust-driven design
The problem
Payroll errors destroy employee trust fast — one wrong cycle and credibility takes months to rebuild. Field visit tracking felt surveillance-heavy. Both modules needed to serve admin efficiency without alienating the people they affected most.
My role
I led design for both modules, running research sessions with HR managers and field employees separately. I worked closely with PM and engineering to define the right data boundaries for location tracking.
Web to mobile
Payroll on web was a dense multi-step process for HR — run configuration, review, approve, publish. On mobile for employees, it became a single payslip view with a plain-language breakdown. For field visits, the web admin view showed maps, routes, and visit logs — the mobile employee view focused on check-in simplicity and transparent data visibility so employees always knew what was being tracked.
Key decisions
For payroll, I designed a pre-payroll review step to catch errors before finalising — reducing errors at source. For employees, a payslip breakdown explained every deduction in plain language. For field visits, I advocated for employee-visible tracking — employees could see exactly what data was shared, which reduced resistance to adoption significantly.
What shipped
A payroll module with pre-run validation, employee-facing payslip breakdowns, and bulk correction flows. A field visit module with transparent location sharing, visit history, and expense linking — used across 10+ industries.
