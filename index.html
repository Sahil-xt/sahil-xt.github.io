Case studies

Four projects from Mewurk — an all-in-one HRMS platform for Indian SMEs used by 500k+ employees. Each tackles a distinct design challenge at the intersection of complexity, trust, and scale.

Case study 01
Attendance & face scan kiosk
Redesigning how 500k+ employees across 1000+ SMEs clock in daily — and building a face-recognition hardware kiosk where zero training was non-negotiable.
Lead designerWeb + MobileHardware kiosk
The problem
HR managers spent hours reconciling attendance records across 15+ status types (present, absent, half-day, error, missed punch, overtime, and more). Employees gamed location-based check-ins. Factory floors needed a touchless solution for workers with low digital literacy — and it had to work without any onboarding or training.
My role
Led UX end-to-end — discovery with HR managers and field employees, shipped mobile flows, the admin web dashboard, and the kiosk interface. Worked closely with PM and hardware engineering throughout.
Key decisions
The kiosk UI was stripped to a single action per screen — large green/red face-frame feedback, iconography and audio cues replacing all text instructions. No onboarding required. Geo-fencing (min 50m radius) was introduced for mobile clock-in, with selfie-based face recognition to prevent proxy attendance. On the admin side, I redesigned the dashboard to surface exceptions first — missed punches, errors, overtime anomalies — rather than raw log tables. For the mobile reminder system, employees set preferred clock-in/out times using a rolling time picker with snooze (10 or 20 min) and dismiss, with edge cases like timezone changes and already-clocked-in states handled gracefully.
Product detail: The attendance system handled 15+ status types — P, A, E, L, WO, P(H), P(W), A½, P½ and more. The admin correction flow let HR update any status with shift-aware time insertion, so "Mark as Present" automatically populated the correct shift in/out times rather than leaving blanks.
What shipped
A face-scan kiosk requiring zero training, deployed across 1000+ SME customers. Mobile attendance with geo-fencing, selfie verification, personalised reminders, and bulk kiosk photo import. A bulk manual correction flow for HR to fix attendance errors at month-end without one-by-one editing.
Case study 02
Leave & holidays management
Leave is one of the most emotionally loaded features in any HRMS. I designed a flexible, configurable system for HR admins and a transparent, trust-building experience for employees.
Lead designerWeb + MobilePolicy-driven UX
The problem
Indian SMEs have highly varied leave policies — different accrual rules per leave type, carry-forward and encashment limits, state-specific holiday calendars, comp-off, short leave, optional holidays, and multi-level approval chains. The existing system was too rigid for admins to configure and too opaque for employees, who often didn't know their balance or why a request was rejected.
My role
Led design for both the admin configuration side and the employee experience. Ran separate research sessions with HR managers and employees to understand the gap between how policies were set and how they were experienced.
Key decisions
The policy builder used progressive disclosure — sensible defaults upfront (annual leave, sick leave, casual leave), with accrual rules, carry-forward limits, and encashment settings behind an "advanced" toggle. I introduced the leave accrual summary view so employees could see every grant and deduction and understand exactly why their balance changed. The team leave calendar showed colleague availability alongside personal balance, reducing back-and-forth with managers. On mobile, the apply flow was redesigned to under 3 taps with inline policy constraints shown in real time — past-date limits, notice period restrictions, and documentation requirements visible before submit, not as errors after.
Product complexity handled: Short leave (a few hours, at start/end/break of day), comp-off with expiry rules, optional holidays requiring manager approval, leave types credited in hours or days, and state-specific regional holidays based on office location — all in a single, coherent leave system.
What shipped
A flexible leave policy builder supporting multiple leave types, accrual rules, carry-forward/encashment, and location-specific holiday calendars. An employee experience with leave accrual summary, team availability calendar, 3-tap apply flow, and real-time policy constraints inline.
Case study 03
Design system & pattern library
As Mewurk scaled to 15+ modules across web and mobile, UI debt was slowing design and engineering. I led the effort to identify, document, and systematise — using Atomic Design principles and Figma's component architecture.
Lead designerWeb + MobileFigma library
The problem
A design team of 4, each working on different modules simultaneously, no shared library, no naming conventions, no documented patterns. Stakeholders kept flagging inconsistencies in demos. The root cause: lack of communication across module teams, each designer using their own components, and no defined design process to follow.
My role
Owned the initiative — ran a team workshop to identify inconsistencies, audited all existing screens by dumping them into one Figma canvas to spot patterns, defined the component architecture, and aligned with engineering on implementation. This ran alongside active product work, not as a separate project.
Key decisions
We adopted Atomic Design (atoms → molecules → organisms → templates → pages), starting by listing all components — input controls, navigation, informational elements — then categorising them correctly. An early mistake was mapping components to the wrong atomic level; after debugging with the team and research, we corrected the categorisation and documented the rules in a wiki. I used Figma's variant feature for component states (default, hover, disabled, error) and nested components for modularity, with version control to track changes. CSS/XML code snippets were generated per component to reduce handoff ambiguity. I also discovered a reusable "Policy Creation" pattern across attendance, leave, holidays, shifts, and WFH — all sharing the same 3-part structure: header → parameters → applicable employees.
Key pattern identified: Policy creation across all 6+ modules followed the same structure — policy header (name + description), policy parameters (module-specific rules), and policy applicable to (employee assignment). Standardising this single pattern eliminated a huge class of inconsistency across the product.
What shipped
A Figma component library with atomic structure, variants, and nested components. A pattern documentation wiki covering interaction patterns, language standards, visual consistency, and IA. 5 consistency pillars defined and acted on. Naming conventions adopted across all new modules. Reduced design-to-handoff time noticeably for new features.
Case study 04
Payroll & field visit tracking
Two modules with the same tension — powerful for admins, anxiety-inducing for employees. The challenge was building trust on both sides while handling real Indian statutory complexity.
Lead designerWeb + MobileTrust-driven design
The problem
Indian payroll involves PF, ESI, TDS, PT, LWF, Form 16, and state-specific statutory rules — all behind an interface used by SME HR managers with no accounting background. One wrong cycle destroys trust. Field tracking, meanwhile, felt like surveillance to employees. Both modules needed to serve admin efficiency without alienating the people they affected most.
My role
Led design for both modules. Ran separate research sessions with HR managers and field employees to understand the trust gap. Worked with PM and engineering to define the right data visibility boundaries for location tracking.
Key decisions
For payroll, I designed a pre-run finalization flow — before exporting, HR sees each employee's pending leave requests, attendance errors, and approved overtime, and can take action to resolve them before the cycle is locked. This reduced errors at source rather than correcting after payout. The employee payslip broke down every deduction in plain language, not accounting shorthand. Role-based access (Admin, HR, Manager, Employee) was mapped to distinct UI states. The payroll export was designed to be compatible with Tally, Zoho, and Saral Pay — requiring careful field mapping UX. For field visits, I advocated for employee-visible tracking — employees could see exactly what location data was shared and when — significantly reducing resistance to adoption.
Payroll finalization flow: Before export, HR reviews pending requests for everyone in the payroll group, views attendance error counts per employee, and can approve/reject requests inline — all before the data is locked. Once exported, the file cannot be modified. This single pre-run step prevented a whole class of after-the-fact payroll corrections.
What shipped
A payroll export module with pre-run validation, role-specific views, plain-language payslip breakdowns, and compatibility with major Indian payroll tools. A field visit module with transparent location sharing, visit history, and expense linking — used across 10+ industries.
Web to mobile — a thread across all projects

Across all modules, I was responsible for translating web designs into the Mewurk mobile app — available on Android 9+ and iOS 12+. Not just resizing: rethinking flows for context, touch, and field conditions.

Rethinking information hierarchy
The web admin dashboard showed dense tables with 40+ fields. On mobile, I distilled it to today's summary, flagged exceptions, and one-tap approvals — what managers need on the go.
Touch-first interactions
Multi-step web flows redesigned for one-handed use — rolling time pickers, bottom sheets, swipe actions, and thumb-reachable primary actions throughout all modules.
Designing for field users
Factory workers and field staff with low digital literacy — every mobile screen had to be self-evident. The kiosk UI took this furthest: no text, no training, just a face in a frame.
Shared token layer
Built a shared token system in Figma so web and mobile referenced the same values — consistent meaning, different rendering. A "primary action" looked right on both platforms.
